Use this reference before writing or revising `design.md`. Produce a concrete implementation contract from inspected repository evidence and approved upstream decisions.

## Ground the Design

Inspect affected code, tests, configuration, dependencies, generated types, and repository instructions before naming paths, symbols, packages, commands, or APIs. Follow installed patterns unless an approved decision changes them.

When the repository does not contain the requested technology yet, verify concrete APIs against current official documentation. Record actual stack-native code only after verifying its version and ownership. Use an explicitly labeled `pseudocode` fence for a preferred contract that cannot be stated as valid native syntax; never disguise guessed syntax as code.

Complete grounding when every named path and external API has repository or official-document evidence.

## Build the Hierarchies

For a feature, include two distinct trees:

1. **Repository Structure:** Show only files and folders created, changed, tested, configured, or generated by the spec. Annotate responsibility, not implementation steps.
2. **Component Hierarchy:** Show runtime composition and ownership from entry point through UI components, services, state owners, and external collaborators.

Keep the trees production-oriented. Split independently owned units, keep collaborating units near each other when repository conventions allow it, and avoid speculative file inventories.

Complete this section when a reader can locate every owned responsibility and explain how the runtime pieces compose.

## Write Native Contracts

Use code or pseudocode as the primary form for schemas and interfaces. Include relevant inputs, outputs, fields, constraints, indexes, relationships, result unions, error types, lifecycle ownership, and security boundaries.

Prefer the selected stack's real vocabulary: Convex validators and indexes, SQL DDL, TypeScript types, REST request/response schemas, GraphQL types, events, CLI signatures, or equivalent native forms. Keep identifiers consistent across components, flows, tables, and tasks.

When data models or operations genuinely do not apply, write `**Not applicable:**` followed by the inspected reason. This is an exception for an absent concern, not permission to omit an unresolved design.

Complete contracts when an implementer can use them without inventing a signature, schema, or error shape.

## Diagram Flows and Strategies

Give every nontrivial request flow, state transition, authorization path, persistence strategy, retry/recovery path, concurrency strategy, cache strategy, or background process its own subsection and Mermaid diagram.

Choose the diagram that matches the behavior:

- Flowchart for ownership and branching.
- Sequence diagram for calls and responses.
- State diagram for lifecycle transitions.
- Entity relationship diagram for material relationships not already clear in native schema code.

Add concise pseudocode beside a diagram when ordering, validation, concurrency, or arithmetic could otherwise be implemented inconsistently. Keep static facts in code or tables rather than manufacturing a diagram.

Complete each flow when the diagram shows the full success and material failure path at the designed boundary.

## Use Tables for Inventories

Use one operation table per category present in the stack: queries, mutations, endpoints, commands, events, jobs, or equivalent. Include operation, input, output, typed errors, and upstream identifiers. Use tables for error handling, testing strategy, and traceability as scaffolded.

Keep algorithms and schemas out of inventory cells; link them by stable names to the code or flow subsection that owns the detail.

## Adapt for Bugfixes

Use the bugfix template when `bugfix.md` exists. Replace full feature trees with the smallest affected caller/ownership hierarchy. Show the root cause and corrected flow, changed native interfaces, changed data models or an evidence-based non-applicability reason, error behavior, regression evidence, and preserved `UB` boundaries.

Avoid redesigning unaffected subsystems. Complete the bugfix design only when the smallest shared fix accounts for every expected and unchanged behavior.

## Review the Design

Before approval, resolve every finding:

1. **Coverage:** Every upstream identifier maps to design and verification.
2. **Repository fit:** Paths, packages, APIs, and commands are inspected or officially verified.
3. **Hierarchy:** Repository and runtime ownership are both concrete and non-duplicative.
4. **Contracts:** Schemas and interfaces use native code or explicit pseudocode.
5. **Flows:** Every nontrivial dynamic strategy has a complete Mermaid diagram.
6. **Inventories:** Operations, errors, tests, and traceability are readable tables.
7. **Consistency:** Names, types, errors, and data flow agree throughout.
8. **Scope:** Every design element serves the focused feature or defect.
9. **Completion:** No placeholder, vague instruction, or unresolved choice remains.

Run the validator and request approval only after every check passes.
